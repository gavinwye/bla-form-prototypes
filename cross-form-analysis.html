1. Executive summary

Analysis of all 23 BLA prototype forms to identify shared data fields, repeated UI components, and opportunities for reuse or pre-fill.

Key finding: applicant identity is asked 23 times

Full name, address, telephone, and email appear in every single form. A citizen account or session-level pre-fill could eliminate this repetition entirely.

Key finding: vehicle identity block appears in 20/23 forms

Registration number, chassis number, and engine number form a consistent trio. A vehicle lookup by registration number could pre-fill all three.

Key finding: 10 distinct design patterns, all consistently applied

One-thing-per-page, Check Your Answers, Confirmation panel, joint-ownership fork, and others are used uniformly. The design system is coherent and ready for component extraction.

4. Component types used

Eleven distinct input component types are used across the 23 forms, all drawn from the alpha.gov.bb design system.

Text input All 23 forms

Single-line text. Used for names, IDs, registration numbers, make, model. Bordered wrapper with teal focus ring.

Textarea All 23 forms

Multi-line text. Used for addresses, descriptions, reasons. Same border/focus pattern as text input but taller.

Checkbox All 23 forms

Used for declaration consent on every form. Also used for registration retention option in sale/transfer forms.

Radio buttons 18 / 23 forms

Custom circular buttons (not native). Used for vehicle category, fuel type, drive type, ownership type, purpose, usage classification. Large tap target (size-12).

Date input (DD / MM / YYYY) 8 / 23 forms

Three separate narrow text inputs. Day (5rem wide), Month (5rem), Year (7rem). Hint text "For example, 27 03 2007". Used for transaction dates, inspection dates, insurance dates.

Select / dropdown 6 / 23 forms

Used for parish (11 values), country of manufacture, use classification. Same border styling as text input.

Number input 6 / 23 forms

Used for engine capacity (cc), number of cylinders, axles, passengers, weight (kg), odometer reading (km), sale price (BDS$).

File upload 5 / 23 forms

Used for insurance certificate, examiner certificate, vehicle photos, manufacturer specs, graphics design files, and signature capture. Styled as a bordered upload zone.

Email input 20 / 23 forms

Styled identically to text input but with type="email". Used for applicant and second-party email addresses.

Tel input All 23 forms

type="tel" with format hint XXX-XXXX. Used for all telephone fields. Styled identically to text input.

Datetime picker 1 form

Appears only in motor vehicle inspection to select preferred examination date and time slot. Native date + time inputs.

5. Design pattern inventory

Ten recurring patterns appear across the 23 forms. Each is consistently implemented.

P1

One thing per page

Every question page asks a single question or presents a single related group of fields. Each page has its own H1 (the question), a form section caption, a Back link, and a Continue button.

All 23 forms GOV.UK Service Manual pattern
P2

Check Your Answers summary list

All 23 forms include a penultimate page presenting all entered answers in a key–value table. Answers are grouped by section under H2 subheadings. Each row has a "Change" link navigating back to that question. The submit button appears at the bottom.

All 23 forms 3-column layout: label | value | Change link Visually hidden text for accessibility
P3

Confirmation panel

After submission, all forms show a teal-background panel (bg-bb-teal-00, white text) containing a reference number placeholder and confirmation heading. Followed by "what happens next" guidance and a link back to the BLA or start page.

All 23 forms Reference number format: HDJ2123F bg-bb-teal-00 panel
P4

Start page with eligibility + what you'll need

Every form begins with an introduction page listing who can apply, an estimated completion time, a "What you will need to share" checklist, and a teal "Complete the online form" start button styled as an anchor tag with button classes.

All 23 forms Start button = teal <a> with button classes Last updated date shown
P5

Joint ownership fork

Four forms (Sale Joint, Transfer Joint, Registration Renewal, Registration Summary) have a dedicated page for a second owner. The joint variant asks the same identity fields (name, address, NRN, phone, email) for a second person. The individual variant skips this section entirely.

sale-of-vehicle-joint transfer-of-vehicle-joint vehicle-registration-renewal vehicle-registration-summary
P6

Business vs individual variant pair

Three pairs of forms cover the same transaction but with different applicant types. The business variant adds a registration number, business name, and authorised person section. The individual variant replaces those with the personal NRN and full name.

duty-free-concession-business / individual declaration-company / sole-trader sale-joint / individual
P7

Declaration page with optional signature

All forms include a declaration at or near the end. Three forms (company, partnership, sole trader declarations) additionally include a signature capture component (file upload or canvas). The declaration statement is static text followed by a required checkbox before the user can proceed.

All 23 forms (checkbox) Signature: declaration-company, declaration-partnership, declaration-sole-trader
P8

Conditional reveal (inline show/hide)

A radio or checkbox selection reveals or hides additional fields on the same page without a page navigation. Implemented via data- attributes and JavaScript. Example: selecting "Yes" to disability reveals a description textarea; unchecking "same as present address" reveals mailing address fields; "Joint ownership" reveals second owner section.

motor-vehicle-inspection vehicle-registration-renewal registration-temporary-stay request-for-graphics
P9

Client-side validation with error summary

On Continue, all required fields are validated. Any errors appear in a red error summary box at the top of the page (listing all errors as links) and inline above each affected field. The input border changes to border-bb-red-00 and aria-invalid="true" is set. The page title is prefixed "Error:".

All 23 forms Error summary at page top Inline red error text per field border-bb-red-00 on invalid input
P10

Persistent chrome: top bar, header, alpha banner, footer

All 23 forms share identical page chrome injected by /prototypes/shared/chrome.js. Top bar (bb-blue-100, "Official government website"), yellow header with GOB trident SVG, blue-10 alpha banner ("This page is in Alpha"), and blue-100 footer. The main content area is constrained to max-width 1200px.

All 23 forms Injected via chrome.js Alpha banner on every page max-w-1200px container

6. Recommendations

R1 – Citizen account pre-fill

Full name, address, telephone, and email appear in all 23 forms. NRN appears in 9. A citizen account or session store would let authenticated users skip the applicant identity pages entirely – reducing completion time and error rates across every service.

Fields to pre-fill: Full name · Address · Telephone · Email · NRN

R2 – Vehicle lookup by registration number

Registration number, chassis number, and engine/motor number appear together in 20 forms. Make, model, year, and colour appear in 10–13 forms. A single vehicle lookup API call (triggered when registration number is entered) could pre-fill the entire vehicle identity block.

Fields to pre-fill: Chassis no. · Engine no. · Make · Model · Year · Colour

R3 – Shared form module library

The six data clusters map directly to reusable form modules: ApplicantIdentityBlock, VehicleIdentityBlock, BusinessDetailsBlock, SecondPartyBlock, DocumentUploadBlock, DeclarationBlock. Building these as shared HTML partials or web components would ensure consistency and reduce maintenance burden as forms evolve.

ApplicantIdentityBlock VehicleIdentityBlock BusinessDetailsBlock SecondPartyBlock DocumentUploadBlock DeclarationBlock

R4 – Standardise NRN collection

NRN is collected in 9 forms but labelled inconsistently: "ID number / National Registration Number", "National Registration Number", "ID number". A single canonical label and validation pattern (YYMMDD-XXXX) should be defined and used uniformly.

Canonical label: "National Registration Number" · Format: YYMMDD-XXXX

R5 – Reduce duplicate individual/joint/business variants

Sale and Transfer forms exist in two variants (individual/joint) with near-identical page sequences. These could be merged into a single form with an early "ownership type" question (single / joint) that conditionally adds the second owner section – halving the form count from 23 to around 17.
